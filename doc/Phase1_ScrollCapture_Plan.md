# Phase 1: 스크롤 캡처 & 이미지 스티칭 도구 — 상세 개발 계획서

## 1. 프로젝트 개요

### 1.1 목적

소스 PC의 HDMI 출력을 캡처카드를 통해 타겟 PC에서 수신하고, 소스 PC에서 문서를 자유롭게 스크롤할 때 캡처된 프레임을 하나의 연속된 문서 이미지로 합성하는 브라우저 기반 도구를 개발한다.

### 1.2 하드웨어 구성

| 구분 | 역할 | 비고 |
|------|------|------|
| 소스 PC | 문서를 열고 스크롤 | 별도 소프트웨어 불필요 |
| 캡처카드 | HDMI 신호 -> USB 변환 | Anyport AP-HDC4K (UVC 호환) |
| 타겟 PC | 프로그램 구동, 캡처 & 조립 | Windows, 브라우저 사용 |

### 1.3 신호 흐름

```
소스 PC (HDMI 60fps) -> 캡처카드 (MJPEG 30fps) -> USB 3.0 -> 타겟 PC (브라우저 30fps 수신)
```

### 1.4 핵심 제약조건

- 타겟 PC에 별도 서버 프로그램 설치 없이, 브라우저에서 HTML 파일을 열어 바로 사용 가능해야 한다.
- 캡처카드는 UVC 호환이므로 브라우저의 `getUserMedia()` API로 웹캠처럼 접근한다.
- 스크롤 방향은 세로(위아래) 및 가로(좌우)를 모두 지원한다.
- **빠른 스크롤, 역방향 스크롤, 왔다갔다 스크롤을 모두 지원한다.**

## 2. 사용자 워크플로우

### 2.1 기본 사용 시나리오

```
1. 타겟 PC에서 브라우저로 컨트롤 패널(HTML) 열기
2. 캡처카드 장치 선택 -> 실시간 미리보기 확인
3. 스크롤 방향 선택 (세로 / 가로)
4. [시작] 버튼 클릭 -> 캡처 모드 진입
5. 소스 PC에서 문서를 자유롭게 스크롤 (빠르게, 왔다갔다 가능)
6. [종료] 버튼 클릭 -> 캡처 중단, 자동 조립 시작
7. 조립 완료 (수 초) -> 완성된 문서 이미지 미리보기 & 다운로드
```

### 2.2 사용 주의사항

- 스크롤 중 화면에 팝업이나 오버레이가 뜨면 결과에 영향을 줄 수 있다.
- 매칭 실패가 연속 발생하면 UI에 경고가 표시된다.
- 캡처 시간이 길어지면 (60초 이상) 프레임 상한에 도달할 수 있다.

## 4. 기술 아키텍처

### 4.1 2단계 파이프라인

전체 동작은 **캡처 단계**와 **조립 단계**로 나뉜다.

```
+-------------------+      +-------------------+
|    캡처 단계       |      |    조립 단계       |
|    (실시간)        | ---> |    (캡처 종료 후)   |
+-------------------+      +-------------------+
|                   |      |                   |
| 30fps 프레임 수집  |      | 절대 위치 맵 생성  |
| 변화 감지 & 필터링 |      | Canvas 크기 결정   |
| JPEG 압축 저장     |      | 프레임 디코딩      |
| Worker 오프셋 계산 |      | Canvas에 그리기    |
|                   |      | 경계 블렌딩        |
| 출력:             |      | 출력:              |
|  frames[] (JPEG)  |      |  완성 이미지       |
|  offsets[] (좌표)  |      |  PNG/JPEG 다운로드 |
+-------------------+      +-------------------+

상세: Sub-Plan 1           상세: Sub-Plan 2
```

- **캡처 단계** 상세: [Phase1_Sub1_Capture.md](./Phase1_Sub1_Capture.md)
- **조립 단계** 상세: [Phase1_Sub2_Assembly.md](./Phase1_Sub2_Assembly.md)

### 4.2 핵심 설계 결정

| 결정 사항 | 선택 | 근거 |
|-----------|------|------|
| UI 렌더링 | `React.createElement` 직접 호출 | 빌드 도구/JSX 변환 불필요, 단일 HTML 유지 |
| 프레임 동기화 | `requestVideoFrameCallback` | 캡처카드 실제 프레임에 동기화 (setInterval 대비 정확) |
| 프레임 저장 | JPEG Blob 배열 | Raw ImageData(2MB) 대비 1/25 크기(80KB), 수백 프레임 저장 가능 |
| 오프셋 계산 | Web Worker + NCC | UI 블로킹 방지, Transferable로 복사 비용 제거 |
| 자유 스크롤 | 절대 위치 맵 + 시간순 덮어쓰기 | 역방향/왔다갔다 스크롤 자연 대응 |
| 조립 시점 | 캡처 종료 후 일괄 | 자유 스크롤 시 실시간 조립 불가, 대신 오프셋은 실시간 계산 |

### 4.3 기술 스택

| 영역 | 기술 | 선택 이유 |
|------|------|-----------|
| UI | React (`createElement`) + Tailwind CSS (CDN) | 빌드 도구 없이 컴포넌트 구조화 |
| 영상 입력 | `getUserMedia()` | 서버 없이 브라우저에서 직접 캡처카드 접근 |
| 프레임 동기화 | `requestVideoFrameCallback` | 캡처카드 프레임 도착에 정확히 동기화 |
| 프레임 압축 | `canvas.toBlob('image/jpeg')` | 메모리 효율적 저장 |
| 오프셋 계산 | Web Worker (인라인, Blob URL) | 별도 파일 없이 단일 HTML 유지 |
| 이미지 매칭 | NCC 자체 구현 (2D 탐색) | 외부 라이브러리 의존 없음 |
| 이미지 디코딩 | `createImageBitmap()` | 비동기 디코딩, 메모리 효율적 |
| 결과 저장 | `canvas.toBlob()` + 다운로드 링크 | 브라우저 네이티브 |

### 4.4 외부 의존성

- 없음. 순수 브라우저 API만 사용하여 서버 및 외부 라이브러리 의존 없이 동작한다.
- React와 Tailwind CSS는 CDN으로 로드한다.
- Web Worker는 인라인(Blob URL)으로 생성하여 별도 파일 없이 단일 HTML에 포함한다.

## 5. 핵심 알고리즘 요약

> 상세 알고리즘은 서브 계획서를 참조한다.

### 5.1 캡처 단계 ([Sub-Plan 1](./Phase1_Sub1_Capture.md))

1. **프레임 수집**: `requestVideoFrameCallback`으로 캡처카드 30fps에 동기화하여 프레임을 수집한다.
2. **변화 감지**: 80x45 썸네일 그레이스케일 비교로 정지 프레임을 제거한다 (임계값 3).
3. **JPEG 저장**: 변화 있는 프레임을 960x540으로 축소 후 JPEG Blob(~80KB)으로 압축하여 배열에 저장한다.
4. **오프셋 계산**: Web Worker에서 연속 프레임 쌍의 NCC 2D 매칭을 수행하여 오프셋(dx, dy)을 실시간 계산한다. Worker 배압 조절로 처리 속도에 맞춰 프레임을 자연 건너뛴다.

### 5.2 조립 단계 ([Sub-Plan 2](./Phase1_Sub2_Assembly.md))

1. **오프셋 체인 구성**: 유효한(confidence >= 0.85) 오프셋만으로 프레임 연결 체인을 구성한다. 실패 구간에서 체인이 단절되면 가장 긴 체인을 사용한다.
2. **절대 위치 맵**: 오프셋을 누적하여 각 프레임의 절대 좌표(x, y)를 계산한다. 음수 좌표는 정규화한다.
3. **Canvas 생성 & 그리기**: 전체 범위 크기의 Canvas를 생성하고, JPEG Blob을 하나씩 디코딩하여 절대 위치에 그린다. 나중 프레임이 이전 프레임을 덮어쓴다.
4. **경계 블렌딩**: 겹침 영역에서 16px 선형 블렌딩을 적용한다.
5. **출력**: 결과 Canvas를 PNG/JPEG로 변환하여 다운로드한다.

### 5.3 메모리 사용량

```
캡처 중:
  JPEG Blob 배열: ~80KB x 프레임 수
    10초 캡처 (~150프레임): ~12MB
    30초 캡처 (~450프레임): ~36MB
    60초 캡처 (~900프레임): ~72MB
  직전 프레임 ImageData: ~2MB
  Worker용 버퍼: ~2MB
  합계: ~16MB ~ 76MB

조립 중:
  결과 Canvas: 가변 (최대 960 x 32,000 x 4 = ~117MB)
  디코딩 중 프레임 1장: ~2MB
  합계: ~119MB (최대)

전체 피크: ~195MB (60초 캡처 + 최대 크기 조립 시)
```

```

|   +-- 스크롤 가능한 이미지 뷰어
```

```javascript
{
  // 장치
  selectedDeviceId: string | null,
  stream: MediaStream | null,

  // 캡처
  isCapturing: boolean,
  frames: Array<{ blob: Blob, frameIndex: number, timestamp: number }>,

  frameCount: number,

  assemblyProgress: number,        // 0~100

  // 결과
  resultImageBlob: Blob | null,
  resultImageUrl: string | null,
  resultInfo: { width: number, height: number, fileSize: number } | null,

  // UI
  statusMessage: string,
  warningMessage: string | null,
  errorMessage: string | null,
}
```

## 7. 개발 단계 (세부 태스크)

### Step 1: 프로젝트 셋업 & 영상 입력

**목표:** 브라우저에서 캡처카드 영상을 실시간으로 표시한다.

**태스크:**
- HTML 기본 구조 생성 (단일 파일, CDN으로 React + Tailwind 로드)
- React.createElement 기반 컴포넌트 구조 작성
- `getUserMedia()`로 캡처카드 접근 구현
- 장치 목록 조회 (`enumerateDevices()`) 및 선택 UI 구현
- `<video>` 태그에 실시간 영상 출력
- 해상도, FPS 확인 및 표시

**완료 기준:** 브라우저에서 캡처카드 영상이 실시간으로 표시된다.

### Step 2: 프레임 캡처 파이프라인

**목표:** 프레임 수집, 변화 감지, JPEG 저장, Worker 통신 파이프라인을 구축한다.

**상세:** [Sub-Plan 1](./Phase1_Sub1_Capture.md) 섹션 2~5

**태스크:**
- 인라인 Web Worker 생성 (Blob URL 방식)
- `requestVideoFrameCallback` 기반 프레임 수집 루프 (`requestAnimationFrame` 폴백 포함)
- 960x540 작업 Canvas + 80x45 썸네일 Canvas 구성
- 썸네일 기반 변화 감지 (그레이스케일 diff, 임계값 3)
- `canvas.toBlob('image/jpeg', 0.90)` 압축 저장
- Worker 배압 조절 (유휴 시 전송, 작업 중이면 대기 프레임 교체)
- 시작/종료 버튼 UI, 스크롤 방향 선택 UI
- 프레임 수/매칭 품질 실시간 표시

**완료 기준:** 시작~종료 사이에 변화 프레임이 JPEG Blob으로 저장되고, Worker에 순차 전달된다.

### Step 3: NCC 오프셋 계산 (Worker)

**목표:** Worker 내에서 두 프레임 간의 스크롤 오프셋을 정확히 계산한다.

**상세:** [Sub-Plan 1](./Phase1_Sub1_Capture.md) 섹션 6~7

**태스크:**
- Worker 내 그레이스케일 변환 + 50% 축소 함수
- 스트립 추출 (세로: 하단 20%, 가로: 우측 20%)
- 템플릿 분산 검사 (저분산 시 스트립 위치 재시도)
- NCC 2D 탐색 (주 방향 전체 + 부 방향 +/-10px)
- Transferable 전송으로 복사 비용 제거
- 오프셋(dx, dy) + confidence 반환
- 역방향 스크롤 대응 (음수 오프셋 정상 반환)

**완료 기준:** 다양한 스크롤 속도/방향에서 정확한 오프셋을 계산한다.

### Step 4: 이미지 조립

**목표:** 캡처 종료 후 프레임들을 하나의 이미지로 합성한다.

**상세:** [Sub-Plan 2](./Phase1_Sub2_Assembly.md)

**태스크:**
- 오프셋 체인 구성 (유효 오프셋 필터링, 체인 단절 처리)
- 절대 위치 맵 생성 (오프셋 누적, 음수 좌표 정규화)
- 결과 Canvas 크기 결정 (최대 32,000px 제한)
- JPEG Blob 순차 디코딩 (`createImageBitmap`) + Canvas 그리기
- 경계 블렌딩 (16px 선형)
- 진행률 표시
- 자유 스크롤 처리 (역방향, 왔다갔다 -> 시간순 덮어쓰기)

**완료 기준:** 다양한 스크롤 패턴에서 하나의 연속된 문서 이미지가 생성된다.

### Step 5: 결과 출력 & 다운로드

**목표:** 완성된 이미지를 미리보기하고 파일로 다운로드할 수 있다.

**태스크:**
- 결과 Canvas 크롭 (실제 콘텐츠 영역만)
- Canvas -> Blob 변환 (PNG, JPEG 선택)
- 다운로드 링크 생성 및 자동 다운로드
- 파일명 자동 생성 (타임스탬프 기반)
- 이미지 정보 표시 (크기 px, 파일 용량, 사용 프레임 수)

**완료 기준:** 결과 이미지를 PNG/JPEG로 다운로드할 수 있다.

### Step 6: 예외 처리 & 안정성

**목표:** 다양한 상황에서 안정적으로 동작한다.

**태스크:**
- 캡처카드 미인식/연결 끊김 처리
- 매칭 실패 연속 발생 시 경고
- 프레임 상한(2,000) 도달 시 자동 종료
- Canvas 크기 초과 시 분할 저장
- `requestVideoFrameCallback` 미지원 브라우저 감지 및 폴백
- 매칭 품질 지표 표시 (평균 NCC, 실패율)
- 체인 단절 시 사용자 안내

**완료 기준:** 에러 상황에서 적절한 피드백을 제공하며 크래시하지 않는다.

## 8. 테스트 계획

### 8.1 단위 테스트 시나리오

| 항목 | 테스트 내용 | 예상 결과 |
|------|-----------|-----------|
| 장치 접근 | USB 캡처카드 getUserMedia 호출 | 영상 스트림 정상 수신 |
| 중복 감지 | 동일 화면에서 프레임 반복 수집 | 변화 없는 프레임 필터링됨 |
| JPEG 저장 | 960x540 프레임 JPEG 압축 | Blob 크기 60~80KB |
| 오프셋 계산 | 100px 아래 스크롤된 두 프레임 | dy ~= 100 |
| 역방향 오프셋 | 위로 스크롤된 두 프레임 | dy < 0 |
| 균일 영역 | 흰 배경의 두 프레임 | 저분산 감지, 스트립 위치 재시도 |
| 절대 위치 맵 | 왔다갔다 오프셋 시퀀스 | 정확한 2D 좌표 계산 |
| 이미지 조립 | 5개 프레임 조립 | 하나의 연속 이미지 생성 |
| Worker 배압 | Worker 지연 시뮬레이션 | 프레임 자연 건너뜀, 최신 유지 |

### 8.2 통합 테스트 시나리오

| 시나리오 | 설명 |
|----------|------|
| 짧은 문서 | A4 2~3페이지 세로 스크롤 -> 조립 |
| 긴 문서 | A4 10페이지 이상 -> Canvas 크기 한계 테스트 |
| 가로 스크롤 | 넓은 스프레드시트 좌우 스크롤 -> 가로 조립 |
| 빠른 스크롤 | 빠르게 스크롤 -> 프레임 건너뜀 동작 및 결과 품질 |
| 역방향 스크롤 | 아래로 갔다가 위로 되돌림 -> 정상 조립 |
| 왔다갔다 스크롤 | 위아래로 반복 이동 -> 중복 영역 정상 처리 |
| 정지 구간 | 스크롤 중 잠시 멈춤 -> 불필요한 프레임 없음 |

## 9. 알려진 제한사항 및 리스크

| 항목 | 설명 | 대응 방안 |
|------|------|-----------|
| Canvas 최대 크기 | ~32,768px 제한 (브라우저별 상이) | 32,000px 제한, 초과 시 분할 저장 |
| MJPEG 아티팩트 | 캡처카드 MJPEG 특성상 프레임 간 미세 노이즈 | 경계 블렌딩으로 완화 |
| NCC 오매칭 | 균일 영역에서 잘못된 오프셋 | 템플릿 분산 검사 + 스트립 위치 재시도 |
| 오프셋 오차 누적 | 다수 프레임 누적 시 드리프트 | 0.85 고임계값으로 완화. 심하면 결과 Canvas 하단과 매칭 (Sub-Plan 1 방법 B) |
| requestVideoFrameCallback 호환 | 비 Chromium 브라우저 미지원 | requestAnimationFrame 폴백 |
| 캡처카드 호환성 | 일부 캡처카드 브라우저 미인식 | UVC 호환 제품 사용 권장 |
| 화면 변화 오인 | 커서 깜빡임 등 | 오프셋 최소값 임계 설정 |
| 극도로 빠른 스크롤 | 30fps에서도 겹침 부족 (10,000px/s 이상) | 매칭 실패 경고 |
| 체인 단절 | 매칭 실패 구간에서 이미지 누락 | 가장 긴 체인 사용, 단절 위치 안내 |
