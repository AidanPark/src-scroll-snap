# Phase 1: 스크롤 캡처 & 이미지 스티칭 도구 — 상세 개발 계획서

## 1. 프로젝트 개요

### 1.1 목적

소스 PC의 HDMI 출력을 캡처카드를 통해 타겟 PC에서 수신하고, 소스 PC에서 문서를 스크롤할 때 캡처된 여러 프레임을 하나의 연속된 문서 이미지로 합성(스티칭)하는 브라우저 기반 도구를 개발한다.

### 1.2 하드웨어 구성

| 구분 | 역할 | 비고 |
|------|------|------|
| 소스 PC | 문서를 열고 스크롤 | 별도 소프트웨어 불필요 |
| 캡처카드 | HDMI 신호 → USB 변환 | Anyport AP-HDC4K (UVC 호환) |
| 타겟 PC | 프로그램 구동, 캡처 & 스티칭 | Windows, 브라우저 사용 |

### 1.3 신호 흐름

```
소스 PC (HDMI 출력) → 캡처카드 (AP-HDC4K) → USB 3.0 → 타겟 PC (브라우저)
```

### 1.4 핵심 제약조건

- 타겟 PC에 별도 서버 프로그램 설치 없이, 브라우저에서 HTML 파일을 열어 바로 사용 가능해야 한다.
- 캡처카드는 UVC 호환이므로 브라우저의 `getUserMedia()` API로 웹캠처럼 접근한다.
- 스크롤 방향은 세로(위↔아래) 및 가로(좌↔우)를 모두 지원한다.

## 2. 사용자 워크플로우

### 2.1 기본 사용 시나리오

```
1. 타겟 PC에서 브라우저로 컨트롤 패널(HTML) 열기
2. 캡처카드 장치 선택 → 실시간 미리보기 확인
3. 스크롤 방향 선택 (세로 / 가로)
4. [시작] 버튼 클릭 → 캡처 모드 진입
5. 소스 PC에서 문서를 처음부터 끝까지 천천히 스크롤
6. [종료] 버튼 클릭 → 캡처 중단
7. 자동으로 스티칭 처리 시작
8. 완성된 하나의 문서 이미지 미리보기 & 다운로드
```

### 2.2 사용 주의사항

- 소스 PC에서 스크롤 시 너무 빠르지 않게 일정한 속도를 유지해야 한다.
- 스크롤 중 화면에 팝업이나 오버레이가 뜨면 스티칭 결과에 영향을 줄 수 있다.
- 문서 영역 외의 UI 요소(스크롤바, 상태바 등)는 스티칭 시 자동으로 처리되어야 한다.

## 4. 기술 아키텍처

### 4.1 전체 구조

```
브라우저 (단일 HTML 파일)
│
├── getUserMedia() API
│   └── 캡처카드(UVC) → MediaStream → <video> 실시간 출력
│
├── Canvas API
│   ├── <video>에서 프레임 스냅샷 추출
│   ├── 프레임 간 변화 감지 (중복 제거)
│   └── 스티칭 결과 이미지 합성
│
├── 스크롤 감지 모듈 (JavaScript)
│   ├── 이전 프레임과 현재 프레임 비교
│   ├── 겹침 영역 탐색 (상관관계 기반 매칭)
│   └── 스크롤 오프셋(dx, dy) 계산
│
├── 스티칭 엔진 (JavaScript)
│   ├── 프레임별 오프셋 누적
│   ├── 겹치지 않는 새 영역만 추출
│   └── 최종 Canvas에 순차적 합성
│
└── 결과 출력
    ├── Canvas → 미리보기 표시
    └── Canvas.toBlob() → 파일 다운로드
```

### 4.2 기술 스택

| 영역 | 기술 | 선택 이유 |
|------|------|-----------|
| UI 프레임워크 | React (단일 JSX 파일) | 상태 관리 편의, 컴포넌트 구조화 |
| 영상 입력 | getUserMedia() | 서버 없이 브라우저에서 직접 캡처카드 접근 |
| 프레임 처리 | Canvas 2D API | 픽셀 단위 이미지 조작 |
| 이미지 매칭 | 수동 구현 (NCC 알고리즘) | 외부 라이브러리 의존 최소화 |
| 결과 저장 | Blob API + 다운로드 링크 | 브라우저 네이티브 |

### 4.3 외부 의존성

- 없음. 순수 브라우저 API만 사용하여 서버 및 외부 라이브러리 의존 없이 동작한다.
- React와 Tailwind CSS는 CDN으로 로드한다.

## 5. 핵심 알고리즘 상세

### 5.1 프레임 캡처 전략

```
캡처 루프 (requestAnimationFrame 또는 setInterval)
│
├── <video>에서 Canvas로 drawImage()
├── Canvas에서 ImageData 추출
├── 이전 프레임과 유사도 비교
│   ├── 유사도 > 임계값 → 동일 프레임, 건너뜀 (화면 정지 상태)
│   └── 유사도 < 임계값 → 변화 감지, 프레임 저장
└── 저장된 프레임 배열에 추가
```

**캡처 간격:** 약 200~500ms (초당 2~5프레임). 문서 스크롤 속도에 맞추어 적절한 겹침이 발생하도록 조정한다.

**중복 판정 방법:** 두 프레임을 축소(예: 160x90)한 뒤 픽셀 차이의 평균을 계산한다. 차이가 임계값(예: 평균 차이 < 5) 이하면 동일 프레임으로 판정하여 저장하지 않는다.

### 5.2 스크롤 오프셋 감지 알고리즘

두 연속 프레임 간의 스크롤 이동량(오프셋)을 구하는 것이 핵심이다.

**방법: 스트립 기반 정규화 교차상관(NCC) 매칭**

```
세로 스크롤의 경우:

프레임 A (이전)          프레임 B (현재)
┌──────────────┐       ┌──────────────┐
│              │       │ ████████████ │ ← B 상단에 A 하단의
│              │       │ ████████████ │   내용이 겹쳐 있음
│              │       │              │
│ ████████████ │       │              │
│ ████████████ │       │              │
└──────────────┘       └──────────────┘
  A의 하단 스트립         B 전체에서 검색

1. A의 하단에서 스트립(높이 = 프레임 높이의 20~30%)을 잘라냄
2. B의 전체 영역에서 이 스트립과 가장 유사한 위치를 탐색
3. 매칭된 위치로부터 수직 오프셋(dy)을 계산
```

**NCC(Normalized Cross-Correlation) 계산:**

```
두 영역 T(템플릿), I(이미지)에 대해:

NCC = Σ(T - T̄)(I - Ī) / sqrt(Σ(T - T̄)² × Σ(I - Ī)²)

결과 범위: -1 ~ +1 (1에 가까울수록 일치)
매칭 임계값: 0.7 이상이면 유효한 매칭으로 판정
```

**성능 최적화:**

- 매칭 전 프레임을 그레이스케일로 변환하여 계산량을 1/3로 줄인다.
- 프레임을 축소(예: 50%)하여 매칭 후, 원본 스케일로 오프셋을 환산한다.
- 세로 스크롤 시 가로 방향은 고정이므로, 수직 방향으로만 1D 탐색한다.
- 가로 스크롤 시 반대로 수평 방향 1D 탐색으로 처리한다.

### 5.3 스티칭(이어붙이기) 알고리즘

```
입력: 프레임 배열 [F0, F1, F2, ..., Fn]
출력: 하나의 긴 문서 이미지

1. 결과 Canvas 생성 (초기 크기 = F0 크기)

2. for i = 1 to n:
     a. Fi-1과 Fi 간 오프셋 (dx, dy) 계산
     b. 매칭 실패 시 → 해당 프레임 건너뜀 (또는 경고 표시)
     c. 매칭 성공 시:
        - 결과 Canvas 크기를 새 영역만큼 확장
        - Fi에서 겹치지 않는 새 영역만 잘라냄
        - 결과 Canvas의 끝에 새 영역을 그림

3. 결과 Canvas를 이미지로 출력

세로 스크롤 예시:
┌──────────┐
│  F0 전체  │
├──────────┤
│ F1 새영역 │  ← F1에서 F0과 겹치지 않는 하단 부분만
├──────────┤
│ F2 새영역 │
├──────────┤
│ F3 새영역 │
└──────────┘
  최종 결과
```

**블렌딩 처리:** 이어붙인 경계선이 눈에 띄지 않도록, 겹침 영역에서 선형 블렌딩(alpha 그라디언트)을 적용한다. 이는 선택적 기능으로, 기본 동작에서는 단순 이어붙이기로 처리하고 결과 품질을 확인 후 필요 시 추가한다.

### 5.4 메모리 관리

브라우저에서 대량의 고해상도 프레임을 메모리에 보관하면 문제가 발생할 수 있다.

- **프레임 저장 시 해상도 축소:** 원본 1920x1080 대신, 스티칭용으로 적절한 해상도(예: 1280x720 또는 960x540)로 축소하여 저장한다.
- **프레임 수 제한:** 최대 프레임 수를 설정하고(예: 200프레임), 초과 시 경고를 표시한다.
- **스티칭 완료 후 프레임 배열 해제:** 결과 이미지가 생성되면 중간 프레임들을 메모리에서 해제한다.
- **최종 이미지 크기 제한:** Canvas의 최대 크기는 브라우저마다 다르지만, 대략 가로 또는 세로 최대 32,768px까지 지원된다. 이를 초과할 경우 분할 저장 옵션을 제공한다.

## 6. 컴포넌트 구조

### 6.1 React 컴포넌트 트리

```
<App>
├── <DeviceSelector />        장치 선택 드롭다운
├── <VideoPreview />           실시간 미리보기 (video + canvas)
├── <ControlPanel />           조작 버튼 및 설정
│   ├── 스크롤 방향 선택 (세로/가로)
│   ├── 캡처 시작 버튼
│   ├── 캡처 종료 & 스티칭 버튼
│   └── 상태 표시 (프레임 수, 진행률)
├── <ResultViewer />           스티칭 결과 미리보기
│   ├── 스크롤 가능한 이미지 뷰어
│   └── 다운로드 버튼 (PNG / JPEG)
└── <StatusBar />              하단 상태바 (메시지, 에러)
```

### 6.2 주요 상태 (State)

```javascript
{
  // 장치
  selectedDeviceId: string | null,
  stream: MediaStream | null,

  // 캡처
  isCapturing: boolean,
  capturedFrames: ImageData[],
  frameCount: number,

  // 설정
  scrollDirection: 'vertical' | 'horizontal',
  captureInterval: number,        // ms 단위 캡처 간격
  captureResolution: number,      // 축소 비율 (0.5 = 50%)

  // 스티칭
  isStitching: boolean,
  stitchProgress: number,         // 0~100
  resultImageBlob: Blob | null,
  resultImageUrl: string | null,

  // UI
  statusMessage: string,
  errorMessage: string | null,
}
```

## 7. 개발 단계 (세부 태스크)

### Step 1: 프로젝트 셋업 & 영상 입력

**목표:** 브라우저에서 캡처카드 영상을 실시간으로 표시한다.

**태스크:**
- HTML/React 기본 구조 생성 (단일 파일)
- `getUserMedia()`로 캡처카드 접근 구현
- 장치 목록 조회 (`enumerateDevices()`) 및 선택 UI 구현
- `<video>` 태그에 실시간 영상 출력
- 해상도, FPS 확인 및 표시

**완료 기준:** 브라우저에서 캡처카드 영상이 실시간으로 표시된다.

### Step 2: 프레임 캡처 기능

**목표:** 시작/종료 버튼으로 프레임 캡처를 제어하고, 변화가 있는 프레임만 저장한다.

**태스크:**
- 시작/종료 버튼 UI 구현
- `<canvas>`에 `drawImage()`로 프레임 스냅샷 추출
- setInterval 기반 주기적 캡처 루프 구현
- 프레임 간 유사도 비교 로직 구현 (중복 제거)
- 캡처된 프레임 수 실시간 표시
- 스크롤 방향 선택 UI (세로/가로 라디오 버튼)

**완료 기준:** 시작~종료 사이에 변화가 있는 프레임만 배열에 저장된다.

### Step 3: 스크롤 오프셋 감지

**목표:** 두 연속 프레임 간의 스크롤 이동량을 정확히 계산한다.

**태스크:**
- 그레이스케일 변환 함수 구현
- 스트립 추출 함수 구현 (세로: 하단 스트립, 가로: 우측 스트립)
- NCC(정규화 교차상관) 매칭 함수 구현
- 오프셋(dx, dy) 계산 로직 구현
- 매칭 실패 처리 (임계값 미달 시 건너뛰기)
- 성능 최적화: 축소 후 매칭, 1D 탐색

**완료 기준:** 천천히 스크롤한 두 프레임에서 정확한 픽셀 오프셋을 계산할 수 있다.

### Step 4: 이미지 스티칭

**목표:** 전체 프레임을 하나의 긴 이미지로 합성한다.

**태스크:**
- 결과 Canvas 동적 크기 조절 구현
- 프레임별 순차 스티칭 로직 구현 (새 영역만 이어붙이기)
- 세로/가로 양방향 스티칭 지원
- 스티칭 진행률 표시 (프로그레스 바)
- 스티칭 결과 Canvas를 화면에 미리보기로 표시

**완료 기준:** 여러 프레임이 하나의 연속된 문서 이미지로 합성된다.

### Step 5: 결과 출력 & 다운로드

**목표:** 완성된 이미지를 미리보기하고 파일로 다운로드할 수 있다.

**태스크:**
- 결과 이미지 스크롤 가능한 미리보기 뷰어 구현
- Canvas → Blob 변환 (PNG, JPEG 선택)
- 다운로드 링크 생성 및 자동 다운로드
- 파일명 자동 생성 (타임스탬프 기반)
- 이미지 크기(px, 파일 용량) 표시

**완료 기준:** 스티칭된 이미지를 PNG/JPEG로 다운로드할 수 있다.

### Step 6: 예외 처리 & 품질 개선

**목표:** 안정성과 사용 편의성을 높인다.

**태스크:**
- 캡처카드 미인식 시 에러 메시지 및 안내
- 스크롤이 너무 빠를 때 경고 (매칭 실패율 모니터링)
- 메모리 사용량 모니터링 및 제한
- Canvas 최대 크기 초과 시 분할 저장
- 캡처 간격 조절 슬라이더 (사용자 설정)
- 캡처 해상도 조절 옵션

**완료 기준:** 다양한 상황에서 안정적으로 동작하며, 사용자에게 적절한 피드백을 제공한다.

## 8. 테스트 계획

### 8.1 단위 테스트 시나리오

| 항목 | 테스트 내용 | 예상 결과 |
|------|-----------|-----------|
| 장치 접근 | USB 캡처카드 getUserMedia 호출 | 영상 스트림 정상 수신 |
| 중복 감지 | 동일 화면에서 프레임 2회 캡처 | 1개만 저장됨 |
| 오프셋 계산 | 100px 아래로 스크롤된 두 프레임 | dy ≈ 100 계산 |
| 스티칭 | 5개 프레임 세로 스티칭 | 하나의 연속 이미지 생성 |
| 가로 스크롤 | 좌우로 스크롤된 프레임들 | 가로로 이어붙인 이미지 생성 |

### 8.2 통합 테스트 시나리오

| 시나리오 | 설명 |
|----------|------|
| 짧은 문서 | A4 2~3페이지 분량 문서를 세로 스크롤 → 스티칭 |
| 긴 문서 | A4 10페이지 이상 문서 → 메모리 및 Canvas 크기 한계 테스트 |
| 가로 스크롤 | 넓은 스프레드시트를 좌우 스크롤 → 가로 스티칭 |
| 빠른 스크롤 | 빠르게 스크롤하여 매칭 실패 유발 → 경고 메시지 확인 |
| 혼합 스크롤 | 세로 스크롤 중 약간의 가로 흔들림 → 정상 처리 여부 |

## 9. 알려진 제한사항 및 리스크

| 항목 | 설명 | 대응 방안 |
|------|------|-----------|
| 스크롤 속도 | 너무 빠르면 프레임 간 겹침 없음 | 매칭 실패 시 경고, 캡처 간격 조절 |
| 브라우저 Canvas 최대 크기 | 약 32,768px 제한 (브라우저별 상이) | 초과 시 분할 저장 |
| 메모리 | 고해상도 프레임 다수 보관 시 메모리 부족 | 프레임 축소 저장, 최대 개수 제한 |
| NCC 매칭 정확도 | JS 구현이 OpenCV 대비 느리고 정확도가 낮을 수 있음 | 축소 후 매칭, 임계값 튜닝 |
| 캡처카드 호환성 | 일부 캡처카드가 브라우저에서 인식 안 될 수 있음 | UVC 호환 제품 사용 권장 |
| 화면 변화 오인 | 커서 깜빡임 등이 스크롤로 오인될 수 있음 | 변화 영역 비율 임계값 설정 |
