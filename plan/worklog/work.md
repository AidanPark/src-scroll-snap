### 2.3 기술 스택 확정 여부

| 항목 | 결정 사항 |
|------|-----------|
| 프레임워크 | React (createElement, JSX 없음) |
| 스타일링 | Tailwind CSS (CDN) |
| 배포 형태 | 단일 HTML 파일, 서버 불필요 |
| OCR 엔진 | Tesseract.js (로컬) + 멀티모달 LLM API (OpenAI, Anthropic, Google) |
| 영상 입력 | getUserMedia → MediaStream (UVC 캡처카드) |
| 이미지 처리 | Canvas API |
| 상태 관리 | React useState/useReducer (외부 라이브러리 없음) |
| 대상 환경 | 1280px+ 데스크톱, 모던 브라우저 |

→ 모든 기술 스택이 확정되어 있고, 외부 의존성이 최소화되어 있음

## 4. 최종 결론

**3개 전송 방안 기술 문서와 UI/UX 기획서 모두 코딩 착수에 필요한 수준의 완성도를 갖추고 있다.**

- 기술 문서: 3개 모두 반복 검토를 거쳐 기술적 오류 0건
- UI/UX 기획서: 2,107줄, 3차 검토 완료, 상태 정의·전이·와이어프레임·컴포넌트 트리 모두 구비
- 기술 스택: 확정, 외부 의존성 최소
- 코딩 중 결정해야 할 사항은 있으나, 설계 수준에서 방향이 제시되어 있어 블로커 없음

**→ 코딩 작업을 시작해도 좋다.**

### 5.6 구현 시 주의사항

- React는 `createElement` 기반(무 JSX), Tailwind CDN, 단일 HTML 파일 제약을 끝까지 유지.
- 상태명/전이명은 문서 표기와 1:1로 맞춰 상태바/사이드바 매트릭스 불일치가 재발하지 않게 관리.
- OCR 엔진 표시는 텍스트 캡처/블록 텍스트/상태바 모두 동일 포맷(`OCR: ...`)으로 통일.
- Cloud Vision은 `DOCUMENT_TEXT_DETECTION` 기능 표기로 유지하고, API Key/OAuth 경로를 모두 고려해 UI와 런타임 분기 구현.
- 메모리 관리(프레임 2장 유지, 스트립 누적, Blob 오프로딩, 예산 경고/자동 중단)는 후순위가 아니라 코어 파이프라인으로 구현.
- 문서에 명시된 경고 조건(과속, 연속 실패, 장치 끊김, 권한 거부)별 사용자 복구 경로를 UI에서 반드시 제공.
